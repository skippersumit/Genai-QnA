## Day 9 – Document Ingestion Pipeline (RAG Foundation)

---

## 🎯 Objective of Day 9

The goal of Day 9 was to build a **complete document ingestion pipeline** that prepares data for Retrieval-Augmented
Generation (RAG).

By the end of Day 9, the system is capable of:

* Accepting a document as input
* Extracting raw text
* Chunking the text into smaller units
* Generating embeddings for each chunk
* Storing embeddings with metadata in a vector database

This pipeline forms the **write path** of a GenAI system.

---

## 🧠 High-Level Flow

```
Client
  |
  v
POST /documents/ingest
  |
  v
Text → Chunks → Embeddings → Vector DB (Qdrant)
```

Each document is transformed into **multiple vectorized chunks**, enabling semantic retrieval later.

## 🧱 Core Components

### 1️⃣ Controller – Entry Point

```java

@PostMapping("/ingest")
public ResponseEntity<Map<String, Object>> ingest(
        @RequestBody Map<String, String> body
) {
    String text = body.get("text");
    String documentId = ingestionService.ingest(text);

    return ResponseEntity.ok(
            Map.of(
                    "documentId", documentId,
                    "status", "INGESTED"
            )
    );
}
```

**Responsibilities:**

* Accept user input
* Delegate orchestration to service layer
* No business logic inside controller

---

### 2️⃣ Service – Ingestion Orchestration

```java
public String ingest(String text) {

    String documentId = UUID.randomUUID().toString();
    List<String> chunks = chunkText(text, 300);

    for (int i = 0; i < chunks.size(); i++) {
        String chunk = chunks.get(i);

        String embeddingJson = embeddingService.embed(chunk);
        float[] vector = EmbeddingParser.extractVector(embeddingJson);

        vectorStoreService.upsert(
                UUID.randomUUID().toString(),
                vector,
                Map.of(
                        "documentId", documentId,
                        "chunkIndex", i,
                        "text", chunk
                )
        );
    }

    return documentId;
}
```

**Responsibilities:**

* Generate a unique document ID
* Chunk text
* Embed each chunk
* Store embeddings + metadata in vector DB

This is the **single source of truth** for ingestion logic.

## 🧠 Chunking Strategy

### Current Approach (Day 9)

* Fixed-size chunking
* 300 characters per chunk
* Deterministic and simple

```java
private List<String> chunkText(String text, int chunkSize) {
    List<String> chunks = new ArrayList<>();
    for (int i = 0; i < text.length(); i += chunkSize) {
        int end = Math.min(text.length(), i + chunkSize);
        chunks.add(text.substring(i, end));
    }
    return chunks;
}
```

### Why Chunking Is Necessary

* LLM context windows are limited
* Smaller chunks improve retrieval precision
* Each chunk becomes independently searchable

---

## 🧠 Embedding Strategy

* Each chunk is embedded **independently**
* Same embedding model used for:

    * Ingestion
    * Querying (later)

### Why this matters

Embeddings from different models live in **different vector spaces** and are not comparable.

## 🧱 Vector Storage (Qdrant)

Each chunk is stored as a **point** in Qdrant with:

### Vector

* 768-dimensional float array
* Generated by `nomic-embed-text`

### Payload (Metadata)

```json
{
  "documentId": "7d79ec59-1e92-41c5-a28f-18f9330d856b",
  "chunkIndex": 0,
  "text": "Spring Boot is a Java framework..."
}
```

**Why payload matters:**

* Allows grouping chunks by document
* Enables traceability and debugging
* Will later support source attribution

## 🧪 Verification & Observations

### Qdrant Collection Status

```json
"points_count": 1,
"indexed_vectors_count": 0
```

### Interpretation (Important)

* `points_count = 1` → vector stored successfully
* `indexed_vectors_count = 0` → expected

**Reason:**
Qdrant builds HNSW indexes **lazily** once vector count crosses a threshold (default ~10,000).
Until then, it performs brute-force search.

This is **normal behavior**, not an error.

## 🧠 Common Pitfalls Encountered (and Fixed)

### 1️⃣ Invalid Vector Serialization

❌ Using `Arrays.toString(float[])`
✅ Using proper JSON serialization via Jackson

### 2️⃣ Invalid Point IDs

❌ Arbitrary strings like `"doc_test_1"`
✅ UUIDs (`UUID.randomUUID()`)

### 3️⃣ Missing Payload in Search

❌ Results appear empty
✅ Use `"with_payload": true`

These are **real-world issues** interviewers care about.

## 🎤 Interview Talking Points (Very Important)

You can confidently explain:

### Why chunk documents?

> To balance semantic precision with context size limits of LLMs.

### Why embed chunks independently?

> Because retrieval happens at chunk granularity, not document granularity.

### Why store metadata with vectors?

> To enable traceability, grouping by document, and source attribution.

### Why is indexing delayed in Qdrant?

> Indexes are built lazily for performance; brute-force search is used for small datasets.

### Why ingestion and query paths are separate?

> Ingestion is write-heavy and offline; querying is read-heavy and latency-sensitive.
